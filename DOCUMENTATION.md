# Документация по проекту WebRTC P2P Phone

Этот документ представляет собой всеобъемлющий обзор приложения WebRTC P2P Phone, включая его архитектуру, детали реализации, а также инструкции по установке.

## 1. Обзор проекта

WebRTC P2P Phone — это веб-приложение, которое позволяет пользователям совершать peer-to-peer аудио/видео звонки с использованием стандарта WebRTC. В приложении реализованы регистрация и аутентификация пользователей, отображение списка онлайн-пользователей и интерфейс для звонков в реальном времени. Обмен сигнальными сообщениями для установки соединения WebRTC происходит через WebSocket, а для обхода ограничений NAT используется TURN-сервер.

## 2. Архитектура

Приложение использует многокомпонентную архитектуру, полностью контейнеризованную с помощью Docker.

*   **Фронтенд:** Одностраничное приложение (SPA) на React и TypeScript. В production-режиме собирается в статические файлы и раздается с помощью собственного Nginx.
*   **Бэкенд:** Приложение на Java Spring Boot, которое обрабатывает бизнес-логику, аутентификацию пользователей и предоставляет WebSocket-сервер для сигнализации WebRTC.
*   **Прокси-сервер Nginx:** Основная точка входа в приложение. Он обрабатывает SSL-терминацию (HTTPS) и маршрутизирует входящие запросы: `/api` и `/signal` на бэкенд, а все остальные — на сервис фронтенда.
*   **TURN-сервер (Coturn):** Критически важный компонент для WebRTC. Coturn используется как STUN-сервер для определения публичных IP-адресов клиентов и как TURN-реле для случаев, когда прямое P2P-соединение невозможно из-за сложных сетевых конфигураций (например, симметричный NAT).
*   **База данных (MariaDB):** Хранит данные пользователей.

## 3. Бэкенд (Java / Spring Boot)

Бэкенд отвечает за управление пользователями, аутентификацию и сигнализацию WebRTC.

*   **Фреймворк:** Spring Boot
*   **Аутентификация:** Spring Security с использованием JSON Web Tokens (JWT)
*   **Обмен данными в реальном времени:** Spring WebSocket
*   **Система сборки:** Maven

### Ключевые пакеты и классы:

*   **`configs`**:
    *   `SecurityConfig.java`: Настраивает Spring Security, включая публичные (`/auth/**`) и защищенные маршруты.
    *   `WebSocketConfig.java`: Настраивает конечную точку WebSocket и обработчики.
    *   `JwtAuthHandshakeInterceptor.java`: Перехватывает запросы на установление WebSocket-соединения для аутентификации пользователей по JWT-токену.
*   **`controllers`**:
    *   `AuthController.java`: Обрабатывает регистрацию и вход пользователей, выпуская JWT при успешной аутентификации.
    *   `WebRTCConfigController.java`: Предоставляет клиентам конфигурацию для `RTCPeerConnection`, включая адреса STUN/TURN серверов и временные учетные данные.
    *   `SocketHandler.java`: Ядро логики сигнализации. Управляет сессиями WebSocket, обрабатывает входящие сигнальные сообщения и пересылает их соответствующему пиру.
*   **`services`**:
    *   `JwtService.java`: Предоставляет утилиты для генерации и проверки JWT.
    *   `UserRegistration.java`: Содержит бизнес-логику для регистрации нового пользователя.

### Процесс аутентификации и получения конфигурации WebRTC

1.  Пользователь входит в систему через REST-эндпоинт `/auth/login`.
2.  Бэкенд возвращает JWT.
3.  Фронтенд сохраняет JWT и запрашивает конфигурацию WebRTC с эндпоинта `/api/webrtc-config`.
4.  `WebRTCConfigController` генерирует временные `username` и `password` для TURN-сервера, используя секрет `TURN_PASSWORD` из `.env` файла, и возвращает их вместе с URL сервера.
5.  Фронтенд использует эти данные для настройки `RTCPeerConnection`.

## 4. Фронтенд (React / TypeScript)

Фронтенд — это современное SPA, обеспечивающее реактивный и отзывчивый пользовательский интерфейс.

*   **Фреймворк:** React
*   **Язык:** TypeScript
*   **Сборщик:** Vite

### Структура проекта:

*   **`components`**: Переиспользуемые компоненты (`VideoPlayer.tsx`, `CallControls.tsx`).
*   **`contexts`**: React Context для глобального управления состоянием (`AuthProvider.tsx`, `SocketProvider.tsx`).
*   **`hooks`**:
    *   `useAuth.ts`: Хук для доступа к контексту аутентификации.
    *   `useSocket.ts`: Хук для доступа к WebSocket.
    *   `useWebRTC.ts`: Ядро логики WebRTC на фронтенде. Он обрабатывает создание `RTCPeerConnection`, управляет медиапотоками и взаимодействует с сигнальным сервером.
*   **`pages`**: Компоненты-страницы для различных маршрутов (`HomePage.tsx`, `LoginPage.tsx`).

### Логика WebRTC

Хук `useWebRTC.ts` является центральным элементом функциональности звонков. Он выполняет следующие шаги:
1.  Получает конфигурацию ICE-серверов от бэкенда.
2.  Получает локальный медиапоток пользователя (`getUserMedia`).
3.  Создает `RTCPeerConnection` с полученной конфигурацией.
4.  Использует `SocketContext` для отправки и получения сигнальных сообщений (offers, answers, ICE candidates).
5.  Обрабатывает ICE-кандидатов для установления P2P-соединения.

## 5. Установка и запуск проекта

Проект полностью контейнеризирован и запускается с помощью Docker Compose. Убедитесь, что вы создали и настроили файл `.env` согласно `README.md`.

### Production-окружение

Этот режим собирает production-ready образы и запускает полный стек сервисов.

```bash
docker-compose -f docker-compose.yml -f docker-compose.prod.yml up --build -d
```

Приложение будет доступно по адресу вашего сервера. Для остановки выполните:
```bash
docker-compose -f docker-compose.yml -f docker-compose.prod.yml down
```

### Среда для разработки (Development)

Этот режим использует `Vite` с горячей перезагрузкой для фронтенда.

```bash
docker-compose -f docker-compose.yml -f docker-compose.dev.yml up --build -d
```

Фронтенд будет доступен по адресу `http://localhost:5173`. Для остановки выполните:
```bash
docker-compose -f docker-compose.yml -f docker-compose.dev.yml down
```
